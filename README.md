# Система управления конференциями

Лёгковесная веб-система для организации научных конференций: сбор заявок и тезисов, формирование программы, публикация сборников. Работает из коробки одной командой `docker run`, поддерживает темы оформления в стиле Nikola.

---

## Возможности

- Создание нескольких конференций с независимыми дедлайнами и настройками
- Приём заявок на доклады и тезисов (PDF/DOCX) от участников
- Модерация заявок и тезисов администратором
- Формирование расписания конференции
- Публикация материалов (программы, сборников)
- Уведомления по почте (опционально)
- Система тем оформления с наследованием (Jinja2, аналогично Nikola)
- Поддержка Markdown при описании конференции
- SQLite по умолчанию, MariaDB/MySQL опционально

---

## Быстрый старт

Доступны два равноправных способа запуска: через `docker run` с готовым образом из Docker Hub или через `docker compose` со сборкой из исходников. Оба запускают один и тот же сервис.

---

## Вариант 1. Через `docker run` (готовый образ)

Не требует клонирования репозитория и сборки. Образ тянется с Docker Hub.

### Минимальный запуск

```bash
docker run -d -p 5000:5000 ваш_ник/confservice
```

Откройте `http://localhost:5000`. Логин администратора и сгенерированный пароль будут напечатаны в логах контейнера — посмотреть их:

```bash
docker logs <container_id> | head -n 30
```

Вывод выглядит так:

```
========================================================================
  АДМИНИСТРАТОР СОЗДАН АВТОМАТИЧЕСКИ

  Логин:  admin
  Пароль: aB3xK9zQ-pLm7vTr

  Сохраните эти данные — пароль больше не будет показан.
  Чтобы задать свои значения, установите переменную окружения ADMIN_DATA.
========================================================================
```

Сохраните пароль — он показывается один раз, при первом запуске. Войти в админку: `http://localhost:5000/auth/login`.

> В этом режиме все данные (БД, загруженные файлы, секретный ключ) живут **внутри контейнера** и теряются при его удалении (`docker rm`). Для постоянной работы используйте схему с volume — см. ниже.

### С сохранением данных

```bash
docker run -d -p 5000:5000 \
  -v confservice_data:/app/data \
  -v confservice_uploads:/app/uploads \
  ваш_ник/confservice
```

Docker создаст именованные тома автоматически. Данные переживают пересоздание контейнера и обновление образа.

### Со своими учётными данными админа

Если не хотите запоминать сгенерированный пароль:

```bash
docker run -d -p 5000:5000 \
  -e ADMIN_DATA="admin:мой_длинный_пароль" \
  -v confservice_data:/app/data \
  ваш_ник/confservice
```

Формат `ADMIN_DATA`: `логин:пароль` (логин ≥ 4 символов, пароль ≥ 8). Несколько админов через запятую: `admin1:pass1,admin2:pass2`.

### Со своей темой оформления

```bash
docker run -d -p 5000:5000 \
  -e ACTIVE_THEME=my-theme \
  -v ./themes:/app/themes \
  -v confservice_data:/app/data \
  ваш_ник/confservice
```

Подробнее про темы — см. раздел [Темы оформления](#темы-оформления) ниже.

### Обновление образа

```bash
docker pull ваш_ник/confservice
docker stop <container_id>
docker rm <container_id>
docker run -d -p 5000:5000 \
  -v confservice_data:/app/data \
  -v confservice_uploads:/app/uploads \
  ваш_ник/confservice
```

Данные в volume сохраняются.

## Вариант 2. Через `docker compose` (сборка из исходников)

Подходит, если хочется собрать образ локально, держать настройки в файле, а не в командной строке, или планируются модификации кода.

### Минимальный запуск

```bash
git clone <ссылка-на-репозиторий>
cd confService
docker compose up -d
```

Compose сам соберёт образ из локального `Dockerfile` и поднимет сервис. Откройте `http://localhost:5000`. Учётные данные админа смотрите в логах:

```bash
docker compose logs -f
```

Все папки (`data/`, `uploads/`, `logs/`, `themes/`) создаются рядом с `docker-compose.yml` и переживают любые `down`/`up`.

### С кастомной конфигурацией

Скопируйте `.env.example` в `.env` и раскомментируйте/измените нужные параметры:

```bash
cp .env.example .env
# отредактируйте .env: ADMIN_DATA, MAIL_*, ACTIVE_THEME и т.д.
docker compose up -d
```

Файл `.env` подключается автоматически — отдельных команд не нужно. Без `.env` сервис тоже работает (все обязательные значения сгенерируются).

> Требуется Docker Compose 2.24+ (декабрь 2023). В более старых версиях нужно либо обновиться, либо вручную создать пустой `.env`.

### Остановить / перезапустить / обновить

```bash
docker compose down              # остановить и удалить контейнер (данные остаются)
docker compose up -d             # запустить снова
docker compose restart           # перезапустить без пересоздания
git pull && docker compose up -d --build   # обновить из git и пересобрать
```

## Что генерируется автоматически

При первом запуске без явных настроек приложение:

| Параметр | Поведение |
|----------|-----------|
| `SECRET_KEY` | Генерируется случайно, сохраняется в `data/.secret_key`. На последующих запусках читается оттуда — сессии и CSRF-токены не сбрасываются. |
| `ADMIN_DATA` | Создаётся админ `admin` со случайным паролем. Учётные данные печатаются в stdout. Сам пароль в БД лежит как хеш (pbkdf2/scrypt). |
| База данных | Создаётся пустая SQLite в `data/conference_service.db`. |

Если вы задали `SECRET_KEY` или `ADMIN_DATA` через `-e` — используются ваши значения, автогенерация не срабатывает.

Если пароль админа потерян — удалите `data/conference_service.db` и перезапустите контейнер: создастся новый админ с новым паролем.

## Конфигурация через переменные окружения

Все параметры задаются через `-e ИМЯ=значение` при `docker run` (или через `env_file` в docker-compose). Полный список — в [`.env.example`](.env.example).

### Часто используемые

| Переменная | По умолчанию | Описание |
|-----------|--------------|----------|
| `SECRET_KEY` | автогенерация | Секретный ключ Flask (≥ 32 символов) |
| `ADMIN_DATA` | автогенерация | `логин:пароль[,логин2:пароль2]` |
| `DATABASE_URL` | `sqlite:///data/conference_service.db` | Можно подключить MariaDB/MySQL |
| `ACTIVE_THEME` | `default` | Имя папки темы из `themes/` |
| `BASE_URL` | `http://localhost:5000` | Используется в ссылках внутри писем |
| `SESSION_COOKIE_SECURE` | `false` | Поставить `true` при работе за HTTPS |

### Почта (опционально)

Без почты система работает полностью — письма нужны только для уведомлений участников.

```bash
docker run -d -p 5000:5000 \
  -e MAIL_ENABLED=true \
  -e MAIL_SERVER=smtp.gmail.com \
  -e MAIL_PORT=587 \
  -e MAIL_USE_TLS=true \
  -e MAIL_USERNAME=your@email.com \
  -e MAIL_PASSWORD=your_app_password \
  -e MAIL_DEFAULT_SENDER=your@email.com \
  -e EMAIL_VERIFICATION_ENABLED=true \
  -v confservice_data:/app/data \
  ваш_ник/confservice
```

### Внешняя БД (MariaDB / MySQL)

```bash
docker run -d -p 5000:5000 \
  -e DATABASE_URL=mysql+pymysql://user:password@db_host:3306/conference_db \
  -v confservice_uploads:/app/uploads \
  ваш_ник/confservice
```

## Темы оформления

Темы хранятся в папке `themes/` и монтируются как volume.

### Структура темы

```
themes/
└── my-theme/
    ├── theme.yaml          # метаданные
    ├── templates/          # переопределяемые шаблоны Jinja2
    │   └── base.html
    └── static/             # переопределяемые стили и скрипты
        └── css/
            └── base.css
```

### theme.yaml

```yaml
name: Моя тема
description: Описание темы
parent: default             # родительская тема (необязательно)
```

Производная тема содержит **только** те файлы, которые нужно переопределить. Остальные подтягиваются из родительской темы автоматически.

### Подключение темы

```bash
docker run -d -p 5000:5000 \
  -e ACTIVE_THEME=my-theme \
  -v ./themes:/app/themes \
  -v confservice_data:/app/data \
  ваш_ник/confservice
```

---

## Резервное копирование

Достаточно сохранить три тома:

```
data/       — база данных, секретный ключ
uploads/    — загруженные файлы
themes/     — пользовательские темы (если есть)
```

Восстановление: смонтировать те же тома при запуске нового контейнера.

## Локальный запуск без Docker (для разработки)

Требования: Python 3.11+

**Windows:**

```powershell
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
python main.py
```

**Linux / macOS:**

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python main.py
```

`.env` не обязателен — `SECRET_KEY` и админ сгенерируются автоматически (`data/.secret_key`, учётные данные в консоль). Если нужны свои значения — скопируйте `.env.example` в `.env` и раскомментируйте нужные строки.
